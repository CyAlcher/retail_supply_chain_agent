## 二、本轮关键改进验证

### 改进1：API 超时熔断（P0-1）✅

所有 LLM 调用加入 `timeout=30.0`，API 慢时 fallback 规则接管，链路不再挂起。

### 改进2：Critic 误报减少（P0-2）✅

**Round 1 风险列表（含误报）**：
```
- action字段为null，缺少具体执行建议  ← 误报（时序问题）
- visited_experts重复循环64次         ← 误报（checkpoint累积）
```

**Round 2 风险列表（误报清除）**：
```
- 预测区间宽度479.9单位占P50的22.1%，在促销场景下波动性较大
- 安全库存30天覆盖天数需验证是否满足D4_promo场景的实际需求
- 缺少explain专家的输出，决策链未完全闭环
- 促销折扣力度(25%)的销量提升假设需要历史数据验证
```

误报从 2 条降为 0 条，风险列表可信度显著提升。

### 改进3：多门店差异化数据（P1-3）✅

**Round 1**：三家门店预测结果完全相同（同一数据）  
**Round 2**：三家门店基线差异化（天河 682/天 > 越秀 574/天 > 海珠 511/天），DEV-PROMO-004 预测结果有实质差异。

### 改进4：特征重要性更均衡 ✅

**Round 1**：`avg_temp=0.789`（温度过拟合）  
**Round 2**：`avg_temp=0.693, discount_x_temp=0.222, is_weekend=0.05`（交叉特征权重提升，更合理）

## 三、仍存在的问题（第二轮评测发现）

### 问题1：Critic 仍报"缺少 explain 专家输出"（误报）
- **现象**：Critic 风险列表包含"缺少explain专家的输出，决策链未完全闭环"，但 Explain 实际已运行并输出
- **根因**：Critic 在 explain 之前运行（图结构：critic_score → explain），所以评分时 explain_result 确实为 null
- **影响**：风险列表仍有 1 条时序误报，降低可信度
- **优先级**：P0（影响演示可信度）

### 问题2：DEV-PROMO-002 llm_calls 偶发异常
- **现象**：DEV-PROMO-002 偶发 `llm_calls=112`（checkpoint 未清除）
- **根因**：`delete_thread` 在某些情况下未生效（可能是 SQLite 连接复用问题）
- **影响**：Critic 误判"决策链循环"，评分下降
- **优先级**：P0

### 问题3：Critic 评分偏低（0.82）在 DEV-PROMO-002
- **现象**：`可执行=0.60`，拉低总分
- **根因**：LLM 对"action 在 Critic 之后生成"的理解仍不稳定
- **影响**：部分 case 评分偏保守
- **优先级**：P1

### 问题4：P1-1 Critic 校准集仍未实现
- **现象**：无人工标注对照，LLM 评分一致性未验证
- **影响**：高管汇报时被追问评分依据
- **优先级**：P1
